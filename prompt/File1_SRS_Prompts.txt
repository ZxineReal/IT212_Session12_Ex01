PROMPT 3 – SINH PHẦN 2: OVERALL DESCRIPTION (MÔ TẢ TỔNG QUAN)
────────────────────────────────────────────────────────────
Tiếp tục với tài liệu SRS eKYC ABC Bank, hãy viết
Phần 2 – Overall Description bao gồm đầy đủ 3 mục:

2.1 PRODUCT PERSPECTIVE (Góc nhìn sản phẩm)
- Mô tả hệ thống eKYC là một module mới tích hợp vào
  hệ sinh thái ngân hàng số ABC Bank.
- Vẽ sơ đồ kiến trúc tổng thể (context diagram dạng text/
  Mermaid) cho thấy eKYC Engine kết nối với: App Mobile,
  Core Banking, Telcos, AML/PEP Database, SMS Gateway,
  Cơ sở dữ liệu Dân cư (C06/RAR).
- Mô tả giao diện hệ thống: User Interface (App Mobile),
  Hardware Interface (Camera, NFC), Software Interface
  (Core Banking API, Telco API), Communication Interface
  (HTTPS TLS 1.3).

2.2 USER CLASSES AND CHARACTERISTICS (Đặc điểm người dùng)
- Lập bảng mô tả tối thiểu 4 nhóm người dùng:
  Khách hàng cá nhân, Giao dịch viên hậu kiểm (Fraud Analyst),
  Quản trị viên hệ thống, Developer/Integration Team.
- Mỗi nhóm cần: Đặc điểm, Kỹ năng kỹ thuật, Quyền truy cập,
  Mức độ ưu tiên.

2.3 CONSTRAINTS (Các giới hạn)
- Giới hạn Pháp lý: Thông tư 16/2020/TT-NHNN (hạn mức 100tr/tháng),
  Nghị định 13/2023/NĐ-CP (bảo vệ dữ liệu cá nhân).
- Giới hạn Kỹ thuật: Thiết bị tối thiểu (Android 8.0+/iOS 14.0+,
  camera 5MP+), chuẩn TLS 1.3, AES-256.
- Giới hạn Nghiệp vụ: Chỉ phục vụ khách hàng cá nhân trong nước,
  giấy tờ tùy thân hợp lệ còn hạn.

Yêu cầu format: Tiêu đề đánh số (2.1, 2.2, 2.3), bảng Markdown,
sơ đồ Mermaid cho Context Diagram.

PROMPT 4 – SINH PHẦN 3: SPECIFIC FUNCTIONAL REQUIREMENTS
────────────────────────────────────────────────────────────
Tiếp tục tài liệu SRS, hãy viết Phần 3 – Specific Functional
Requirements với đầy đủ 4 module bắt buộc dưới đây.

Với MỖI yêu cầu chức năng, hãy trình bày theo bảng có cột:
| FR-ID | Tên chức năng | Mô tả chi tiết | Input | Output | Priority |

MODULE 3.1 – ĐĂNG KÝ TÀI KHOẢN (Account Registration)
- FR-1.1 đến FR-1.4: Nhập SĐT/Email, gửi OTP, kiểm tra OTP,
  kiểm tra SĐT đã tồn tại.

MODULE 3.2 – UPLOAD & ĐỌC CCCD (OCR Module)
- FR-2.1 đến FR-2.5: Phân biệt loại giấy tờ, trích xuất OCR,
  kiểm tra hạn CCCD, kiểm tra chất lượng ảnh, chống giả mạo
  giấy tờ.

MODULE 3.3 – XÁC THỰC KHUÔN MẶT (Biometrics / Face Auth)
- FR-3.1 đến FR-3.3: Liveness Detection (Active & Passive),
  Face Matching 1:1 với ngưỡng >= 80%, Confidence Score.

MODULE 3.4 – KÍCH HOẠT TÀI KHOẢN (Account Activation)
- FR-5.1 đến FR-5.4: Tạo hợp đồng điện tử, ký OTP, tích hợp
  Core Banking (tạo CIF, cấp tài khoản), gửi SMS thông báo.

Yêu cầu bổ sung:
- Gộp thêm Module Đối soát (FR-4.1 đến FR-4.3): AML/PEP,
  Telco API, Device Fingerprinting.
- Đánh mức Priority: Must Have / Should Have / Nice to Have.
- Tổng số FR tối thiểu: 15 yêu cầu.

PROMPT 5 – SINH PHẦN 4: NON-FUNCTIONAL REQUIREMENTS
────────────────────────────────────────────────────────────
Tiếp tục tài liệu SRS, hãy viết Phần 4 – Non-Functional
Requirements với đầy đủ các tiêu chuẩn sau:

4.1 SECURITY (Bảo mật dữ liệu cá nhân)
- Mã hóa In-transit: HTTPS, TLS 1.3, SSL Pinning.
- Mã hóa At-rest: AES-256 cho PII, ảnh CCCD, dữ liệu
  sinh trắc học.
- An toàn ứng dụng: Chống dịch ngược (Obfuscation), chặn
  chụp màn hình trong luồng eKYC, chặn thiết bị Root/Jailbreak.
- Tiêu chuẩn: PCI-DSS, An toàn thông tin cấp độ 3 (TT 06/2023).

4.2 PERFORMANCE (Thời gian phản hồi API)
- OCR processing time: < 1.5 giây.
- Face Matching + Liveness Check: < 2.0 giây.
- Toàn bộ luồng eKYC (backend, không tính thao tác user): < 10 giây.
- Throughput: 200 TPS bình thường, scale đến 1,000 TPS peak.

4.3 AVAILABILITY (Tính sẵn sàng hệ thống)
- SLA Uptime: >= 99.9% (tức < 8.76 giờ downtime/năm).
- Kiến trúc: Active-Active tại 2 DC độc lập (DC & DR).
- Microservices + Kubernetes + Auto-scaling.
- Không có Single Point of Failure (SPOF).

4.4 USABILITY (Trải nghiệm người dùng – nếu có)
- Toàn bộ luồng eKYC hoàn thành trong < 5 phút.
- Hướng dẫn trực quan từng bước (hình ảnh/animation).

4.5 MAINTAINABILITY (Khả năng bảo trì)
- Logging tập trung (ELK Stack hoặc tương đương).
- Cập nhật ngưỡng tham số không cần re-deploy.

Yêu cầu: Trình bày dưới dạng bảng Markdown có cột:
| NFR-ID | Nhóm | Tiêu chí | Chỉ tiêu cụ thể | Phương pháp kiểm chứng |

GHI CHÚ VỀ KỸ THUẬT ĐẶT PROMPT (PROMPT ENGINEERING NOTES)
==============================================================
Chuỗi 7 prompt trên được thiết kế theo nguyên tắc:

1. ROLE PROMPTING    : Prompt 1 gán vai trò chuyên gia cụ thể
                       (Senior BA + Technical Writer) để định
                       hướng tone, chuyên môn và chuẩn mực
                       output (IEEE 830).

2. CONTEXT INJECTION : Toàn bộ BRD từ Bài 01 được nạp vào
                       ngay từ Prompt 1 giúp AI có đủ dữ liệu
                       nghiệp vụ để sinh SRS chính xác và
                       không bịa đặt thông tin.

3. STRUCTURED OUTPUT : Mỗi prompt yêu cầu output theo cấu
                       trúc rõ ràng (mục số, bảng Markdown,
                       mã Mermaid) để dễ dàng tổng hợp
                       thành tài liệu cuối cùng.

4. SECTION CHAINING  : Mỗi prompt kế thừa (chain) kết quả
                       của prompt trước ("Tiếp tục tài liệu
                       SRS...") để AI duy trì ngữ cảnh và
                       tránh mâu thuẫn nội bộ giữa các phần.

5. CONSTRAINTS       : Ràng buộc format output chi tiết
                       (đánh số FR-x.x, bảng có đủ cột,
                       mã Mermaid render được, ngôn ngữ
                       Tiếng Việt chuyên nghiệp).

6. SPECIFICITY       : Mỗi prompt liệt kê CỤ THỂ những gì
                       cần có (số lượng FR tối thiểu, số
                       thuật ngữ tối thiểu, tên các Use Case)
                       để tránh output mơ hồ hoặc thiếu sót.

7. SYNTHESIS PROMPT  : Prompt 7 là "Assembly Prompt" – yêu
                       cầu AI gộp tất cả các phần đã sinh
                       thành một tài liệu hoàn chỉnh duy nhất,
                       có trang bìa và mục lục.
